# 삶의 부캐 (LIFE Alts) — 개발 시작 문서

> 오늘 한 일이 캐릭터를 키우고, 삶에 따라 직업이 변하는 자기계발 RPG 앱
> 환경: Windows · VS Code · React Native + TypeScript · AI로 개발

---

## 0. 한눈에 보는 범위

**1단계 (MVP, 만든다)**
- 오늘 한 일 기록 (위젯에서 톡톡 + 앱 내 입력)
- 스탯 6개 + 금화 누적
- 직업 판정 (스탯 조합 → 직업, 삶에 따라 변함)
- 캐릭터 표시 (단순 2D + 직업 의상 레이어)
- 데일리 퀘스트 (완수=성장 연출 / 미완수=스탯 잠듦)
- 메인 퀘스트 (장기 목표, 선택)
- 로컬 저장 (서버 없음)

**나중 (2단계+)**
- 직업 대폭 확장(조합·숨은·전직), 금화 소비처 확대
- 친구/공유
- 정교한 검증(기기 연동 등)
- 수익화(프리미엄 의상·테마 등)

> 위치/GPS 없음 → 예전 위치 앱 대비 가장 어려운 부분이 사라져 구현이 단순해짐.

## 1. 기술 스택 (확정)

**React Native + TypeScript** (게임보다 "앱"에 가까우므로 게임 엔진 아님)

| 영역 | 선택 | 비고 |
|---|---|---|
| 프레임워크 | React Native + TypeScript | iOS/AOS 동시, AI 지원 최강 |
| 시작 형태 | Expo (세팅 쉬움) | 위치 안 쓰니 managed로도 출발 가능. 위젯 붙일 때 dev build |
| 로컬 DB | AsyncStorage(간단) 또는 SQLite | 데이터 가벼우니 AsyncStorage로 시작 가능 |
| 캐릭터 | react-native-svg | 단순 2D, 페이퍼돌(베이스+의상 레이어). 우리가 만든 SVG 활용 |
| 위젯 | iOS WidgetKit / 안드로이드 Glance | 톡톡 입력. 네이티브 일부 필요 |
| 알림 | OS 로컬 알림 | 데일리 퀘스트 리마인드 등. 서버 푸시 아님 |
| 색/배경 | **시간대별 색 세트** (아침·낮·노을·밤) | 기기 시각에 따라 배경 전환. 밤하늘 고정 아님 |

**윈도우 주의**: 안드로이드는 윈도우에서 끝까지 가능 → 1차 타깃 안드로이드. iOS 빌드는 Mac 필요 → EAS Build(클라우드) 또는 나중에 Mac.

## 2. 데이터 스키마

서버 대비 필드(id/synced 등)는 가볍게만 유지. 1단계는 로컬.

**확장 대비 (기획서 15장 원칙)**: 나중에 경험 도감·칭호·친구·동기화가 붙을 수 있게 스키마를 여유 있게. 특히 — (1) 직업/스탯은 저장하지 말고 actions에서 매번 파생 계산(판정 규칙 바꿔도 과거 데이터 안 깨짐). (2) actions에 type/tag 필드를 두어 나중에 새 활동 유형(경험류) 추가 가능하게. (3) 스탯은 6개+금화로 고정 — 추가 안 함.

### actions (한 일 기록)
| 필드 | 타입 | 비고 |
|---|---|---|
| id | string | 고유 ID |
| ts | ISO datetime | 한 시각 |
| type | enum | exercise/study/social/create/rest/health/work |
| stat | enum\|null | 흘러들 스탯(work는 금화) |
| amount | number | 무게(물1잔=작게, 운동=크게) |
| note | string\|null | 한 줄 질적 기록(보너스) |
| questId | string\|null | 연결된 퀘스트(있으면) |

### character (캐릭터 상태)
| 필드 | 타입 | 비고 |
|---|---|---|
| stats | {str,int,cha,dex,wis,health} | 현재 능력치(감쇠 반영) |
| gold | number | 누적 금화 |
| job | string | 현재 판정된 직업 |
| streak | number | 연속 기록 일수 |
| cosmetics | object | 적용된 외형/의상 |

### quests (퀘스트)
| 필드 | 타입 | 비고 |
|---|---|---|
| id | string | |
| kind | enum | daily / main |
| label | string | "운동 30분" 등 |
| targetStat | enum | 완수 시 흘러들 스탯 |
| progress / goal | number | 진행/목표 |
| done | bool | |
| period | string | daily: 날짜 / main: 기간 |

## 3. 핵심 로직

### 스탯 감쇠 & 직업 판정 (앱의 심장)
- **롤링 윈도우**: 최근 N일(예: 7일) actions만으로 현재 스탯 계산.
- **감쇠**: 최근 활동일수록 가중치 큼(예: 하루 지날 때마다 ×0.85). → "무게중심 이동"으로 직업이 흐르듯 변함.
- **직업 판정**: 가장 높은 스탯(들) → 단일/조합 직업. 금화 압도 → 상인 계열.
- **업무는 직업을 결정하지 않음**: work는 금화·기본 양분만. 직업은 "일 외 활동"이 가른다. (핵심 원칙)

### 완수/미완수 (전투 아님 — 성장/잠듦)
- 퀘스트 완수 → 캐릭터 성장 연출(스탯·경험치·금화·스트릭, 캐릭터 반응).
- 미완수 → 적(몬스터)이 커지는 게 아니라, 해당 스탯이 천천히 잠듦. "요즘 전사 기운이 잠들었어요"(비난 아닌 관찰).
- 거짓 체크는 자기기만이므로 검증 불필요(명예제).

### 캐릭터 렌더 (페이퍼돌)
- 베이스(몸·얼굴) SVG 고정 + 직업별 의상 레이어를 위에 겹침.
- 직업 바뀌면 의상 레이어만 교체. 직업 추가 = 의상 한 벌 추가.

|---|---|
| 홈(캐릭터) | 캐릭터 중앙 + 현재 직업 + 스탯 요약 + 오늘 퀘스트 진행 |

## 5. 개발 순서 (쉬운 것부터, AI와)

1. **데이터 레이어**: actions/character/quests 타입 + 로컬 저장 + 읽기/쓰기.
2. **오늘 기록 화면**: 태그 톡톡 → action 저장. (가장 단순, 흐름 검증)
3. **스탯 계산 + 직업 판정 로직**: 감쇠·윈도우·조합 규칙.
4. **홈/캐릭터 화면**: 캐릭터 SVG(페이퍼돌) + 직업별 의상 + 스탯 표시.
5. **데일리 퀘스트**: 받기-수행-완수 사이클. 체크 → 성장 연출·보상. (미완수는 스탯 잠듦)
6. **메인 퀘스트**: 장기 목표 + 데일리 자동 제안.
7. **위젯**: 톡톡 입력(2×2), 현황(2×4). (네이티브 일부, 다소 까다로움 → 뒤)
8. **로컬 알림**: 퀘스트 리마인드.
9. **꾸미기/금화 소비처**: 후순위. 반응 보고.

> 1~4까지면 "캐릭터가 내 활동으로 자라고 직업이 바뀐다"는 핵심을 폰에서 확인 가능. 위젯·꾸미기는 그 뒤.

## 6. AI 개발 황금 규칙

1. **잘게 쪼개기**: "앱 만들어줘" ✕ → "오늘 기록 화면만", "직업 판정 함수만" ○.
2. **하나씩 확인**: 받으면 폰에서 돌려보고 → 되면 다음, 안 되면 에러를 AI에게.
3. **SVG/프로토타입을 입력으로**: 대화에서 그린 캐릭터·시간대 배경 SVG를 "RN으로 옮겨줘"의 출발점으로.
4. **어려운 건 뒤로**: 위젯은 7번. 그 전에 RN에 익숙해지기.
5. **실기기 테스트**: 특히 위젯·알림.
6. **버전 확인**: 라이브러리·Expo·OS 정책은 자주 바뀌니 착수 시점 최신 문서 확인.

## 7. 아트 리소스 (AI 생성 시 주의)

- 캐릭터는 **단순 2D + 의상 레이어(페이퍼돌)**. 스타일을 단순하게 할수록 유리.
- **AI 이미지 생성의 약점 = 일관성**: 같은 베이스에 맞는 의상 50벌을 일관되게 못 뽑음.
  - 대응: (1) 스타일을 극단적으로 단순화해 보정 쉽게, 또는 (2) AI는 컨셉/시안용, 실제 레이어는 벡터로 직접/도구로.
- 3D는 비권장: RN 구현 부담↑, AI의 3D 생성·정합성 아직 약함. "앱"이므로 2D로 충분.
- SVG로 만들면 저작권 부담 적고(직접 그린 단순 도형), 색·수정 쉬움.

## 9. 전체 데이터 흐름 (설계도)

지금까지의 부품들이 하루 동안 어떻게 맞물려 도는지. 핵심은 **두 갈래 구조**.

```
① 사용자 행동 (루틴 도장 · 퀘스트 기록)
        ↓
② actions 저장 (로컬) — {활동, 스탯, 무게, 시각}
        ↓
   ┌────┴─────────────────┐
   ↓ 즉시(그날)            ↓ 누적(최근 14일)
 퀘스트 완수 판정          스탯 계산 (감쇠 ×0.9/day)
   ↓                        ↓
 보상 연출                직업 판정 (70% 규칙)
 (경험치·금화·             ↓
  스트릭·캐릭터 반응)      직업 바뀌면 → 전환 연출
   └────┬─────────────────┘
        ↓
③ 화면 표시 (홈 · 직업·스탯 · 퀘스트 · 기록)
```

### 핵심 포인트
- **하나의 데이터(actions)에서 두 흐름이 갈라진다**:
  - **즉시 흐름**: 그날의 행동 → 즉각 보상. "매일의 동기부여". 동기화 불필요, 바로 계산.
  - **누적 흐름**: 최근 14일 행동 → 스탯/직업. "시간을 둔 변화". 매 기록 시 또는 앱 열 때 재계산.
- **즉시 보상 ≠ 직업 변화**: 오늘 운동 한 번은 즉시 경험치를 주지만(즉시), 직업이 전사로 바뀌려면 2주간 누적돼야 함(누적). 이 분리가 "매일 재미 + 장기 변화"를 동시에 만듦.
- **계산 시점**: 스탯/직업 판정은 매번 actions를 다시 훑어 계산(저장값이 아니라 파생값). 윈도우·감쇠 때문에 시간이 지나면 같은 데이터도 결과가 달라짐 → 앱 열 때마다 재계산이 자연스러움.
- **로컬 우선**: 1단계는 전부 기기 내 계산. actions만 잘 쌓으면 나머지(스탯·직업·화면)는 다 파생.
